## 1. 実装レビュー(src/EspBle.cpp / EspBle.h)

### High(初回リリース前に修正すべき)

**H1. worker taskがbusy flagクリア後に`impl`へ書き込む — `end()`とのuse-after-free競合**
- `gattTaskEntry`(EspBle.cpp:694-699)は`gattOperating=false`にした**後**に`impl->pushGattResult(result)`を呼ぶ。`sendTaskEntry`(EspBle.cpp:867-912)も`sending=false`にした後に`queueServerSendResult()`を呼ぶ。
- `end()`(EspBle.cpp:3417-3444)はこれらのflagがfalseになるまでspinしてから`delete impl_`(EspBle.cpp:3454)するため、flagクリアとpushの間のwindowでloop taskが先へ進むと、worker taskが解放済み`EspBleImpl`(mutex/queue)へアクセスする。
- 修正: `connectTaskEntry`(EspBle.cpp:530-539)や`discoveryTaskEntry`と同様に、**resultをpushしてからflagをクリアする**順序へ統一する。
- 検証状態: コード確認済み(レビュー本体+二重確認)。

**H2. 接続lifecycleイベントがNotificationと同じ8-slot queueを共有し、あふれると黙って消える**
- `pushEvent`(EspBle.cpp:212-223)はtail-drop方式で、戻り値は全呼び出し元で無視。容量は`ConnectionEventQueueCapacity=8`(EspBle.cpp:43)。GATT Notificationも同じqueueに入る(`queueNotification` EspBle.cpp:462-481)。
- `update()`の間に8件を超えるNotificationが来ると、**新規のConnected/Disconnected/SecurityChanged/GattResultが落ちる**。`onDisconnected`が発火しないと`hidKeyboardHost_.handleDisconnected()`(EspBle.cpp:4020付近からのみ呼ばれる)も走らず、HID Hostにstale slotとdangling pointerが残り、全releaseイベントも出ない。`droppedEvents`(EspBle.cpp:709)は非公開で観測不能。
- 修正方針: lifecycleイベントとデータイベントのqueue分離、またはlifecycleイベントの優先保持(drop対象をNotificationに限定)+drop数の公開+overflowイベント(CORE_DESIGN 72行の方針どおり)。

**H3. `BLEClient`がCentral接続のたびにleakする(接続失敗時も) — backend調査で確定**
- `connectTaskEntry`(EspBle.cpp:520-541)は試行ごとに`BLEDevice::createClient()`するが、失敗時もdeleteせず、切断時も`removeClientConnection`(EspBle.cpp:286-297)はslotを消すだけでclientを解放するコードがファイル内に存在しない。
- 再接続を繰り返すKeyBridge用途(本ライブラリの主用途)でheapが単調減少する。keyboard不在時の接続retryループでも1回ごとにleak。
- backend確認結果(3.3.10同梱BLE): `createClient()`はstaticな`m_pClient`を上書きするだけで一覧管理はなく、`deinit(false)`は**最後に生成した1個だけ**を解放する(BLEDevice.cpp:1113-1116)。`deleteClient()` APIは存在しない。解放手段はアプリ側の`delete client`のみで、`~BLEClient`はlink切断+最大2秒のevent待ち後にremote service treeを解放する。なおアプリがdeleteした後に`deinit()`が呼ばれると`m_pClient`の**double-delete**になる罠もある(EspBle側で最後のclientをdeleteしない設計が必要)。また`m_connectedClientsMap`が`CONFIG_BT_NIMBLE_MAX_CONNECTIONS`(=3)に達すると`createClient()`はnullptrを返す。

**H4. HID Hostが保持する`BLERemoteCharacteristic*`の無効化が「落ちうるイベント」に依存 — backend調査で緩和(ただしH3修正と連動)**
- `Connection::inputReport/outputReport`(EspBle.cpp:1101-1102、設定はEspBle.cpp:1518-1520)のクリアは`handleDisconnected`(EspBle.cpp:3063-3105)のみ。これはH2のqueueを生き延びたDisconnectedイベント+`update()`呼出しが前提。
- backend確認結果: 切断イベントでは`m_servicesMap`は解放されず、remote characteristicポインタは**即座にはdangling化しない**(解放は`~BLEClient`または`getServices()`再呼出し時のみ)。現状はclientをdeleteしない(=H3のleak)ため偶然UAFを免れているが、**H3を「切断時にclientをdeleteする」形で修正すると本項が現実のUAFになる**。修正はセットで行う(delete前にslotのポインタをクリアする順序保証)。
- 残る実害: イベントdrop時に`ready()`がtrueを返し続け、stale slotが4枠しかないHID Host slotを恒久的に占有し、`setKeyboardLeds`が死んだlinkへのwriteを試みる。
- 補足(backend罠): 同一clientで再接続すると`m_haveServices`が残っているため`getService()`が**切断前の古いhandleのobjectをrediscoveryなしで返す**。client使い回し方式へ変えるなら再接続後の明示的な`getServices()`(全旧ポインタ無効化)が必須。

### Medium

- **M1. worker task内のremote GATT objectのTOCTOU**: `gattTaskEntry`/`discoveryTaskEntry`は冒頭で`isConnected()`を1回確認するだけで、複数ATT往復にわたりservice treeを触る。途中切断でNimBLE host task側が解放・変更する競合に対する同期がない。
- **M2. `disconnect()`のguardがHID Host操作をカバーしない**: EspBle.cpp:3533は`gattOperating && gattConnectionId == id`のみ拒否するが、`discover()`(EspBle.cpp:2823-2828)と`setKeyboardLeds()`(EspBle.cpp:2888-2895)は`gattConnectionId`を設定せず`gattOperating`だけ立てる。HID Discovery中の同一接続disconnectが素通りし(M1を増幅)、逆に無関係な接続のdisconnectが古いIDとの偶然の一致でブロックされうる。
- **M3. Central側MTUが接続時の1回しか取得されない — backend調査で大幅緩和**: backendの`connect()`は通常MTU交換完了(`BLE_GAP_EVENT_MTU`)まで待ってから返り、`getMTU()`は`ble_att_mtu()`のlive値を返すため、接続直後のsnapshotは通常正しい。ただし(a) MTUイベントがtimeout(connect timeout+1秒)までに来ない場合は23がありうる、(b) client側にMTU変更callbackが存在しないため接続後の変化は追えない。Lowへ格下げし、制約として文書化で足りる。
- **M4. passkey表示イベントの接続対応付けが推測**: `queuePasskeyDisplayed`(EspBle.cpp:373-403)は「最初の未暗号化slot」を選ぶ。複数同時Pairingで誤った接続を報告しうる(DECISIONS Security #8の既知課題の具体化)。callback signatureでconn handleを捨てている点が根因。
- **M5. 6KROのphantom/rollover report(ErrorRollOver `0x01`×6)を「全release」として解釈**: `queueInputReport`(EspBle.cpp:1340-1347)は`usage >= 0x04`でfilterするため、7キー以上同時押しで全keyのrelease→再pressの嵐が発生する。標準的なHostはrollover reportを無視する。
- **M6. Report Map判定がバイト列の完全一致**: `supportsSixKeyKeyboard`(EspBle.cpp:1147-1159)は`05 01 09 06 a1 01`等の literal 連続一致を要求する。実在keyboardは項目順序が異なる(`75 08 95 06`等)ことが多く、正当な6KRO keyboardを拒否する。また購読対象は「最初のnotify可能なtype-1 Report」(EspBle.cpp:1468-1472)で、Consumer Control併載keyboardで誤ったreportを購読しうる。**市販keyboard相互運用(P0-4)の最有力障害候補。**
- **M7. 長さ≠8のInput Reportを黙って破棄**(EspBle.cpp:1330-1333): エラーもカウンタもなく「Discovery成功なのにキーが来ない」形で現れる。
- **M8. Advertisingで16-bit Service UUIDごとに別の「Complete List」AD構造を追加**(EspBle.cpp:1765-1774): 同一typeのAD構造の複数出現はCSS Part A §1.1違反。2個目以降のUUIDを見ないscannerがある。
- **M9. HID DeviceのGATT属性に暗号化permissionがない**(EspBle.cpp:2540-2589): `security.enabled`でも`*_ENC/*_AUTHEN` flagなし。HOGPはHID Serviceに暗号化(Security Mode 1 Level 2)を要求しており、insufficient-encryptionエラーが返らないとOS側のPairing誘発も起きず、キー入力が平文で流れうる。**セキュリティ観点で要優先対応。**
- **M10. CCCD購読状態を見ずに全Peripheral接続へnotify — backend調査で裏付け**: `sendInputReport`(EspBle.cpp:2654-2674)/`setBatteryLevel`(EspBle.cpp:2720-2727)は`ble_gatts_notify_custom()`を全接続へ呼ぶ。この関数はCCCDを見ない生送信で、同梱wrapper自身も`m_subscribedVec`で購読済みpeerに絞ってから呼んでいる(BLECharacteristic.cpp:1091-1167)。EspBle側でも購読状態のgateが必要。
- **M11. 切断時の全releaseイベント自体がqueue満杯で落ちうる**(EspBle.cpp:3094-3103): slot消去後にdropされると再生成不能で、stuck key防止という必須動作(HOST_SPEC)が破れる。
- **M12. 2回目の`begin()`が異なるconfigを黙って無視して成功を返す**(EspBle.cpp:3255-3259)。

**#4 raw service登録の順序制約(重要)**: 同梱backendは`CONFIG_BT_NIMBLE_DYNAMIC_SERVICE`無効でビルドされており、
- `BLEDevice::createServer()`は`ble_gatts_reset()`を呼ぶため、**それ以前にraw登録したserviceは消える**。

## 3. P0未確定API事項の分析と推奨

### 3-1. P0-1: ESP32KeyBridge利用形からのAPI確定

1. **`update()`必須化 → 明示`update()`を最終仕様として確定を推奨。**
   - Arduinoのloop駆動モデルに合致し、callback re-entrancy問題を避けられる。KeyBridge adapterが`bridge.update()`から駆動できることはPeer検証済み(DECISIONS HID Host #8)。内部task配送はユーザーcallbackの実行context(stack size・排他)の問題を持ち込み、選択式は初回リリースの複雑さに見合わない。
   - 条件: 「`update()`を呼ばない限りconnect/discoverの完了通知も来ない」ことの明記と、queue overflow可観測化(H2/L2)をセットで。

2. **HID Discovery自動化 → 初回リリースは明示`discover()`維持を推奨、自動化optionは後送り。**
   - 再接続でSecurity完了後に呼ぶ必要があり(DECISIONS HID Host #11)、自動化はSecurity状態機械との結合を持ち込む。adapter境界に自動接続を含めない方針とも一貫。
   - ただし**「いつ呼ぶか」の規範が現状食い違っている**: HOST_SPECは「Pairing/Bonding完了後」、KeyboardHost.ino:49-52は`onConnected`内で即時呼出し。security有効構成では`onSecurityChanged`成功後に呼ぶ形へexampleを揃えるか、「暗号化前Discoveryも動作する」ことを仕様として明記するか、どちらかに統一する(未確認事項§1-8とも関連)。

3. **`setKeyboardLeds()` → profile helperとして維持しつつWrite Without Response優先へ変更を推奨。**
   - 現実装(EspBle.cpp:2920-2921)は`canWrite()`ならwrite with responseを選び、応答往復の間呼び出しtaskをblockする。さらに`gattOperating`排他のため`discover()`実行中は失敗する。
   - HOGPのLED OutputはWWR対応が期待できるため、WWR優先にすれば同期`bool`のままblockしないfire-and-forget helperにできる。汎用GATT側の非同期Resultへの統一は不要(KeyBridgeのLockState転送用途にはfire-and-forgetで十分)。`gattOperating`との競合条件は文書化する。

### 3-2. P0-2: 共通API未確定事項

- **Result型 / operation ID**: 「受理時の同期エラーは`bool`+`lastError*()`、完了は各イベントのerror/detailフィールド」という現行役割分担をDECISIONSへ明文化して確定を推奨。operation IDはCentral GATT同時1件制限が続く間は不要(導入見送りで確定してよい)。なお`lastError_`は単一グローバル状態のため、複数taskからの同時操作で競合する点は文書化(単一loop taskからの呼出しを前提とする旨)。
- **byte container**: pointer+length主軸+`String`便宜overloadの現状を初回リリース仕様として確定し、将来の値型(`EspBleBytes`等)への移行余地を未確定事項として残す表明が現実的。ただし§1-未確認-3(`String`のNUL切り詰め)の確認が先。
- **event queue**: 容量のcompile-time設定(`#define`)+各queueへの`dropped*Count()` getter+overflowイベントを初回リリース前に揃えることを推奨(CORE_DESIGN 72行の方針と現実装の乖離解消。H2/M11/L2への対処と同一作業)。
- **listener API一般化**: HID Host以外への拡大は見送りで確定してよい(DECISIONS HID Host #10の利用例駆動方針どおり)。
